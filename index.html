╔══ FRESH EDITOR :: i686 BUILDS ══╗

Automated builds of Fresh IDE for older 32-bit x86 processors. Fresh does not publish pre-built Linux binaries and its toolchain assumes a modern x86-64 host. Statically linked via musl + zig, no external dependencies. Built nightly, released automatically.

INSTALL

► Quick Install

Download the installer and run it. CPU variant is detected automatically.

$ one-liner

curl -fsSL https://raw.githubusercontent.com/vapenyk/fresh-i686-build/main/fresh-install.sh \
  -o /tmp/fi && sh /tmp/fi install

What the installer does

1. Reads /proc/cpuinfo and suggests the right binary for your CPU.

2. Asks where to install (default: ~/.local/bin).

3. Downloads the binary, makes it executable, renames it to fresh.

4. Downloads config.json to ~/.config/fresh/config.json.

5. If the install dir is not in $PATH, shows you the line to add.

Commands

Command Description
fresh-install.sh install Detect CPU, download binary + config
fresh-install.sh update Update binary and/or config to latest
fresh-install.sh remove Remove binary, config, and metadata
fresh-install.sh status Version, variant, config path, PATH check

Detect your variant manually

grep -ow 'pni' /proc/cpuinfo | head -1

Prints pni → use k8-sse3. Nothing → use pentium4.

Build flags

Flag Value Why
LTO fat Full link-time optimization across all crates
codegen-units 1 Single unit for maximum inlining
panic abort No unwinding code, smaller binary
embed-bitcode yes Required for fat LTO
align-all-functions 4 (16 bytes) Instruction fetch alignment for old CPUs
-s strip Strip debug symbols
linker musl + zig Fully static, no libc dependency

CONFIG

► Low-End Config

Installed to ~/.config/fresh/config.json. The goal is not to strip functionality but to prevent resource runaway on machines with limited RAM. All visual features are kept at defaults.

What is changed and why

Setting Value Why
recovery_save_interval 30s Constant disk I/O every 2s is noticeable on slow HDDs
auto_save_enabled false Recovery saves still protect your work
terminal.jump_to_end_on_output false Prevents forced redraws while scrolling
horizontal_scrollbar false Fewer UI elements to render
LSP process_limits per server rust-analyzer alone can use 500MB+ without limits

LSP memory limits

Server Max RAM Max CPU Status
rust-analyzer 512 MB 50% enabled
gopls 256 MB 50% enabled
typescript-language-server 256 MB 50% enabled
intelephense (PHP) 256 MB 50% enabled
marksman (Markdown) 128 MB 25% enabled
typescript (separate) — — disabled

Disable an LSP you don't need

{ "lsp": { "rust": { "enabled": false } } }

Raise memory limit

{
  "lsp": {
    "rust": {
      "enabled": true,
      "process_limits": { "max_memory_mb": 1024, "max_cpu_percent": 100 }
    }
  }
}

Add Python LSP

{
  "lsp": {
    "python": {
      "command": "pylsp", "args": [], "enabled": true,
      "process_limits": { "max_memory_mb": 256, "max_cpu_percent": 50 }
    }
  }
}
UPDATE & REMOVE

► Update

./fresh-install.sh update

Checks installed version against latest release. Asks separately whether to update the binary and the config.

► Remove

./fresh-install.sh remove

Removes the binary, asks separately about the config, cleans up metadata. Does not touch PATH entries in your shell config.

► Metadata location

~/.local/share/fresh-installer/install.meta

Plain text with three fields: VARIANT, VERSION, INSTALL_DIR.

► Automatic upstream tracking

Checks for new upstream releases every night at 00:00 UTC. Both variants are built and published automatically. Run update any time to pull it.
